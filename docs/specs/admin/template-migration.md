# SDD: поэтапная миграция визуального шаблона Admin

## 1. Цель

Постепенно заменить текущие черновые экраны `apps/admin` визуальной системой и компоновкой из `D:\RiderProjects\template`, сохранив фирменные цвета Kedr, существующую React/MUI-архитектуру, маршрутизацию и контракты API.

Шаблон является визуальным и UX-референсом. Его Next.js-код, демонстрационные данные, изображения, тексты Devias и внешние ссылки не переносятся без отдельной задачи.

## 2. Контекст

| Область     | Текущее состояние                            | Роль в миграции                                                 |
| ----------- | -------------------------------------------- | --------------------------------------------------------------- |
| Admin       | React 19, MUI 9, React Router, Nx            | Целевое приложение                                              |
| Тема Admin  | `apps/admin/src/app/theme/admin-theme.ts`    | Единственная точка настройки MUI-токенов Admin                  |
| Каркас      | `libs/admin/core/shell/src/admin-layout.tsx` | Базовый layout всех защищённых страниц                          |
| Авторизация | `libs/admin/core/auth/src/auth-session.ts`   | Используется для выхода; UI не получает учётные данные напрямую |
| Шаблон      | `D:\RiderProjects\template`                  | Источник структуры, размеров, состояний и визуальных решений    |

## 3. Неперемещаемые границы

- Фирменные цвета из `libs/shared/theme/src/lib/design-tokens.json` и цветовые значения `admin-theme.ts` не меняются.
- `primary.main` остаётся `#18b125`; semantic-цвета `info`, `success`, `warning`, `error` сохраняют текущие значения и назначение.
- Не изменяются API-клиент, `auth-session.ts`, Redux store, конфигурация окружения, глобальная маршрутизация storefront и Angular-приложение.
- Не добавляются зависимости только ради точного копирования шаблона. Для иконок использовать уже установленный `@mui/icons-material`.
- Не копировать данные, бренд, изображения и промо-блоки шаблона.
- Визуальная миграция не создаёт фиктивные бизнес-функции: страницы без утверждённого API должны иметь явно обозначенное статическое или empty-состояние.

## 4. Целевая архитектура

### Библиотеки и ответственность

| Расположение                                  | Ответственность                                                                    |
| --------------------------------------------- | ---------------------------------------------------------------------------------- |
| `apps/admin/src/app/theme`                    | Тема MUI, palette, typography, shadows и component overrides                       |
| `libs/admin/core/shell`                       | Layout, desktop/mobile-навигация, верхняя панель, профиль и breadcrumbs            |
| `libs/admin/dashboard/feature`                | Экран dashboard и его композиция                                                   |
| Новый домен Admin при появлении бизнес-логики | Страницы customers, settings, account по правилам `docs/architecture/admin-sdd.md` |
| `libs/admin/*/ui`                             | Переиспользуемые строго доменные визуальные компоненты                             |

Не создавать общий UI-kit до появления как минимум двух подтверждённых потребителей. Первую реализацию каждого элемента держать рядом с доменом; извлечь в `ui` только при реальном повторном использовании.

### Навигация и маршруты

Целевые пути, если соответствующая feature реализована:

| Путь                                               | Назначение                      | Статус на первом этапе                |
| -------------------------------------------------- | ------------------------------- | ------------------------------------- |
| `/`                                                | Dashboard                       | заменить текущий черновой экран       |
| `/products` или утверждённый текущий путь каталога | Товары/каталог                  | сохранить существующий публичный URL  |
| `/customers`                                       | Клиенты                         | новый маршрут только вместе с feature |
| `/settings`                                        | Настройки рабочего пространства | новый маршрут только вместе с feature |
| `/account`                                         | Профиль текущего пользователя   | новый маршрут только вместе с feature |

`AdminLayout` остаётся родительским route element с `Outlet`. Активность пункта меню определяется через `useLocation`, а переходы выполняются `Link` из `react-router-dom`; API Next.js (`next/link`, `usePathname`, `useRouter`) не используется.

## 5. Визуальные требования

### 5.1 Тема MUI

В `admin-theme.ts` поэтапно перенести из шаблона:

- breakpoints: `xs 0`, `sm 600`, `md 900`, `lg 1200`, `xl 1440`;
- типографическую шкалу и веса, адаптировав только к текущему семейству `Inter, "Segoe UI", Roboto, Helvetica, Arial, sans-serif`;
- массив `shadows` как правила elevation;
- `shape.borderRadius: 8` и локальные радиусы компонентов из overrides;
- overrides для `MuiAvatar`, `MuiButton`, `MuiCard`, `MuiCardHeader`, `MuiCardContent`, `MuiLink`, `MuiStack`, `MuiTab`, `MuiTableBody`, `MuiTableCell`, `MuiTableHead` только после проверки применимости.

Значения palette из шаблона (`neonBlue`, Devias neutral и т. п.) не переносятся. При необходимости нейтральных уровней применять существующие `background`, `text`, `divider`, `secondary` или согласованные новые нейтральные токены без изменения фирменных цветов.

Ручное переключение светлой/тёмной/системной схемы сохраняется через MUI с `cssVariables.colorSchemeSelector: 'data'`.

### 5.2 Каркас

| Элемент           | Требование                                                                                                                                       |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| Desktop sidebar   | Фиксированный слева, ширина `280px`, виден от `lg`; логотип/название Kedr, список маршрутов, active/hover/disabled состояния                     |
| Верхняя панель    | Sticky, высота не менее `56px` (целевой референс `64px`), кнопка мобильного меню, доступные действия, профиль и переключатель темы               |
| Контент           | Сдвинут на ширину sidebar от `lg`, `Container maxWidth="xl"`, вертикальный отступ страницы `64px` или согласованный эквивалент                   |
| Mobile navigation | Временный `Drawer`, ширина до `320px`, идентичные desktop-навигация и active-состояние, закрытие по Escape, backdrop и выбору внутреннего пункта |
| Профиль           | Кнопка-аватар с `aria-label`; popover содержит имя/почту только при наличии данных сессии, переходы в account/settings и действие logout         |
| Breadcrumbs       | На страницах глубже dashboard отображают путь и текущий заголовок; последний элемент не является ссылкой                                         |

Нижние промо-блоки sidebar/mobile-nav из шаблона не входят в scope.

### 5.3 Визуальные компоненты

| Компонент           | Минимальный контракт                                                                                                                          |
| ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| Карточка статистики | Заголовок, значение, иконка, опциональный тренд; не содержит запросы данных                                                                   |
| Таблица             | `Card`-контейнер, горизонтальная прокрутка, шапка, строки hover, empty/loading/error-состояния, пагинация только при наличии контракта данных |
| Toolbar             | Заголовок страницы, breadcrumbs, основные и вторичные действия; на малой ширине действия переносятся без потери доступа с клавиатуры          |
| Профиль             | Отображает данные текущего пользователя, не хранит токены и не вызывает HTTP напрямую                                                         |
| Breadcrumbs         | Принимает типизированный список сегментов `{ label, to? }`; не вычисляет бизнес-названия из URL самостоятельно                                |

## 6. План поставки

Каждая фаза — отдельная reviewable-задача, отдельная спецификация в `docs/specs/admin/...` и отдельная проверка.

### Фаза 0 — фиксация референса

1. Снять скриншоты текущих light/dark экранов и референсных экранов шаблона.
2. Зафиксировать список переносимых MUI overrides и список исключённых элементов.
3. Уточнить реальные маршруты и наличие API для customers, settings и account.

**Приёмка:** есть согласованный список экранов, порядок фаз и отсутствие спорных фирменных цветов.

### Фаза 1 — foundation темы

1. Внести breakpoints, typography, shadows, shape и применимые component overrides в `admin-theme.ts`.
2. Сохранить все current palette values в light и dark schemes.
3. Проверить контраст текста, focus и disabled states в обеих схемах.

**Приёмка:** фирменный зелёный и semantic-палитра не изменились; `System`, `Light`, `Dark` работают и сохраняются; карточки, кнопки и таблицы имеют согласованные radius/elevation.

### Фаза 2 — shell и навигация

1. Разделить текущий `AdminLayout` на небольшие компоненты внутри `admin/core/shell` без изменения public API layout.
2. Внедрить sidebar, sticky top bar и responsive mobile drawer.
3. Подключить навигационную конфигурацию только к существующим маршрутам.
4. Добавить profile popover, используя `logout` из `@admin/core/auth`; данные профиля показывать только из утверждённого источника.

**Приёмка:** desktop и mobile navigation доступны с клавиатуры, active route подсвечен, logout очищает сессию, текущие маршруты не ломаются.

### Фаза 3 — общие визуальные примитивы

1. Реализовать toolbar, breadcrumbs и статистическую карточку в наименьшей подходящей доменной библиотеке.
2. Реализовать табличную оболочку с типизированными props и обязательными состояниями данных.
3. Применить компоненты на уже существующих страницах без подмены бизнес-данных.

**Приёмка:** нет дублирования layout-кода на страницах; все интерактивные и icon-only controls имеют доступные имена.

### Фаза 4 — dashboard

1. Заменить черновой `DashboardPage` на dashboard-композицию из статистических карточек и утверждённых блоков.
2. Для отсутствующих метрик показывать informative empty state, а не числа из шаблона.
3. Подключить реальные данные только через доменный data-access слой.

**Приёмка:** dashboard визуально следует референсу, не содержит Devias-текстов и фиктивных бизнес-показателей.

### Фаза 5 — customers

1. Создать feature только после утверждения контракта списка клиентов и прав доступа.
2. Реализовать filter toolbar, table, selection и server-side pagination в соответствии с API.
3. Добавить URL-параметры фильтра/страницы, если это согласовано с продуктом.

**Приёмка:** поиск, выбор и пагинация не теряют состояние непредсказуемо; пустая, загрузочная и ошибочная выборки различимы.

### Фаза 6 — settings и account

1. Создать settings для подтверждённых настроек рабочего пространства.
2. Создать account для данных текущего пользователя и разрешённых операций профиля.
3. Формы выполнять через существующий подход Admin; сообщения об ошибках направлять через текущий notification provider.

**Приёмка:** форма не отправляется дважды, ошибки доступны скринридеру, секреты и токены не выводятся в UI или логи.

## 7. Сценарии

1. **Навигация desktop.** Пользователь на широком экране выбирает пункт sidebar и видит соответствующую страницу с активным пунктом и breadcrumbs.
2. **Навигация mobile.** Пользователь открывает меню, выбирает внутренний маршрут; drawer закрывается, фокус не остаётся на скрытом элементе.
3. **Смена темы.** Пользователь выбирает режим в верхней панели; цвета бренда остаются текущими, а выбор сохраняется после обновления страницы.
4. **Профиль.** Пользователь открывает avatar popover, переходит в account/settings или завершает сеанс. При отсутствии профиля интерфейс не показывает вымышленные имя и email.
5. **Табличные данные.** Пользователь видит loading, empty, error либо data-state и может пользоваться таблицей без горизонтального нарушения layout на mobile.

## 8. Проверка для каждой фазы

- Запустить точечные тесты изменённых библиотек и добавить тесты только к новой интерактивной логике.
- Выполнить `npx nx lint admin` и lint затронутых библиотек.
- Выполнить `npx nx build admin`.
- Ручная проверка: `320px`, `768px`, `1200px`; light/dark/system; keyboard navigation; visible focus; отсутствующие, загрузочные и ошибочные данные.
- Сравнить экран с шаблоном по структуре, отступам, radius, elevation и typography, но не по его palette.

## 9. Риски и открытые решения

- В `auth-session.ts` пока нет публичного контракта текущего пользователя; до его появления profile UI должен быть нейтральным либо скрывать персональные поля.
- Для customers/settings/account не найдены утверждённые API-контракты. Их реализация начинается после отдельной SDD-спецификации домена.
- Точный состав нейтральной шкалы для сложных states должен быть согласован, если текущих `background/text/divider/secondary` недостаточно.
- Шаблон использует Next.js и сторонний набор иконок; переносить следует поведение и визуальные правила, а не исходный код.
