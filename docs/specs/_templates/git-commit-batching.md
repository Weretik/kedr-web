# SDD: поділ поточних змін на коміти

## Мета

Взяти всі незакомічені зміни поточного робочого дерева та створити кілька
невеликих логічно пов’язаних комітів у наявному стилі репозиторію.

## Межі

- Включити лише зміни, які користувач доручив закомітити.
- Не змінювати вміст файлів без потреби для перевірки або виправлення явної помилки форматування.
- Не змішувати незалежні області: конфігурацію, документацію, інфраструктуру й окремі можливості.
- Не створювати commit, push, branch або PR без явного доручення користувача.

## Порядок роботи

1. Показати `git status --short`, staged/unstaged diff і останні коміти.
2. Скласти групи змін із зрозумілою метою кожного коміту.
3. Виконати доступні точкові перевірки змінених областей.
4. Додавати файли до index окремими групами та створювати окремі commit.
5. Використовувати стиль повідомлень із `git log`, зокрема Conventional Commits.
6. Перевірити `git status --short`, `git diff --check` та підсумкову історію.

## Правила повідомлень комітів

- Використовувати повне повідомлення у форматі Conventional Commits:
  `<type>(<scope>): <чіткий опис завершеної зміни>`.
- Обирати точний `type`: `feat` для функціональності, `fix` для виправлення,
  `docs` для документації, `refactor` для внутрішньої перебудови, `test` для
  тестів, `build` або `chore` для інструментів і залежностей.
- Указувати `scope`, коли зміна належить конкретному застосунку, бібліотеці або
  модулю, наприклад `admin`, `dashboard`, `products` або `git`.
- Описувати результат і предмет зміни, а не дію над файлами: `build dashboard
overview`, а не `update dashboard files`.
- Повідомлення має бути конкретним, чесним і самодостатнім: із нього має бути
  зрозуміло, що саме потрапило до коміту, без перегляду diff.
- Не використовувати нечіткі назви на кшталт `update`, `changes`, `fixes`,
  `wip`, `final` або `misc` без уточнення предмета та результату.
- Не заявляти в повідомленні про те, чого коміт фактично не містить. Якщо
  зміна часткова, прямо зазначати це: `feat(admin): add catalog page shell`.
- Писати опис у наказовому способі, з малої літери, без крапки наприкінці;
  перевага надається англійській мові, якщо це відповідає історії репозиторію.
- Для великого коміту або коміту з кількома пов’язаними частинами обов’язково
  додавати тіло повідомлення після порожнього рядка. У тілі коротко перерахувати
  основні завершені зміни, нові залежності, міграції або важливі обмеження.
- Заголовок залишається стислим (орієнтовно до 72 символів), а деталі великої
  поставки переносяться до тіла. Не замінювати тіло загальними фразами на кшталт
  `various improvements`.

Приклад великого коміту:

```text
feat(admin): build dashboard overview

Add KPI cards, task-progress indicator, sales bar chart and traffic pie chart.
Include recent orders and products sections.
Use MUI X Charts and update the dashboard feature tests.
```

Приклади:

- `feat(admin): add responsive application shell`
- `fix(admin): keep top-bar icons visible in dark mode`
- `docs(admin): document dashboard migration phase`
- `build(admin): add MUI X Charts dependency`

## Критерії приймання

- [ ] Робоче дерево чисте або залишені зміни явно зазначені як виключення.
- [ ] Кожен коміт має одну логічну мету та зрозуміле повідомлення.
- [ ] Кожне повідомлення містить конкретний тип, за потреби scope і повний опис результату.
- [ ] Великі або багатокомпонентні коміти мають інформативне тіло повідомлення.
- [ ] Незалежні зміни не об’єднані в один великий коміт.
- [ ] У звіті вказані результати та невиконані перевірки.
